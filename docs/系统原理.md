# 系统原理 · 答辩备稿(团队复习用)

> 用途:全队对着这份过一遍,答辩(viva)时每个人都能讲清楚。技术名词保留英文。

---

## 0. 一句话

我们用软件模拟一座水厂(digital twin),用软 PLC 控制它,数据经 MQTT 上送、存进时序库、在多个看板显示;一个 Claude AI 助手实时检测故障并给操作员建议,报警通过 HMI、ThingsBoard 云、Telegram 三路推送。

对应作业的工业栈:`field → PLC → MQTT → historian → dashboard → AI`。

---

## 1. 数据流(每 0.5 秒一个 PLC 扫描周期)

```
①物理仿真前进 → ②PLC算控制 → ③叠加操作员/故障 → 打包tags → 发布MQTT
                                                          ├→ historian → InfluxDB → Grafana
                                                          ├→ AI agent → 检测 → Claude建议 → 发回MQTT
                                                          ├→ HMI后端 → 网页轮询显示+控制
                                                          ├→ tb_bridge → ThingsBoard云
                                                          └→ telegram → 手机推送
```

**核心:各组件只通过 MQTT 解耦**,互不直接调用。加新下游(云、Telegram)不用改已有代码。

---

## 2. 三种控制逻辑(LO2,必考)

软 PLC(`app/plc.py`)一次扫描里同时跑:
1. **on/off**:进水泵——液位低于设定开、到上限关(带回差)。
2. **PID**:加药泵——目标 pH=7.0,带抗积分饱和(积分项限幅 0~1)。
3. **状态机**:`FILLING(灌到4.0m) → TREATING(处理15s) → DISCHARGING(放到1.0m) → 循环`。

> 出水泵只在 DISCHARGING 阶段自动开——这是状态机逻辑,不是 bug。

物理模型(`app/process.py`):液位用质量平衡 `dL/dt=(进水-出水)/面积`;pH 用一阶滞后逼近"进水pH+加药提升"。

---

## 3. 四个故障 · 注入与检测(LO4,必考)

注入机制:HMI按钮/CLI → 发 MQTT `control/fault` → sim 的 `FaultManager` 开关 → 每扫描周期在数据流不同环节"动手脚"。

| 层 | 故障 | 怎么注入 | 怎么检测(`app/agent_detector.py`) |
|---|---|---|---|
| Sensor | 液位卡死 | 冻结发布的 LIT101 读数 | **解析冗余**:实测不动但质量平衡预测在动 |
| Equipment | 泵转无流量 | 强制进水流量=0,泵反馈仍"运行" | **指令vs效果**:cmd+反馈ON 但流量≈0 |
| Process | pH越限 | 进水pH降到1.5,PID救不回 | **设定值偏差**:\|pH-7\|>1.0 持续 |
| Infra | 数据中断 | sim 停止发布 | **数据陈旧**:>5s 无任何数据 |

**为什么先规则后AI**:规则快且确定,保证 60s 内报警;AI 慢但负责解释。AI 掉线也照样报警。

## 4. AI 助手 · 真·实时推理(LO5,必考)

- 流程(`app/agent_main.py` + `agent_assistant.py`):detector 检出新故障 → 把"故障+实时tag快照"发给 Claude → 返回 `{severity, diagnosis, actions, reasoning}`。
- **怎么证明是真实时,不是预设**:
  1. 回答**引用当前实时数值**(如 pH 5.83、level 22.3%、dose 1.0);
  2. 会**交叉核对多个传感器、发现矛盾**(如"流量读数0但阀开泵转水位涨→可能传感器故障");
  3. 有几秒**API延迟**;
  4. **杀手锏**:拿掉 API key → 退回很短的离线预案;装上 → 长篇引用实时数据。
- **相同情况→相同回答是正常的**(安全建议不该随机)。
- **调试故事(加分)**:我们曾发现回答总一样,排查出是 `max_tokens` 太小导致 Claude 长回答被截断、JSON 解析失败而**默默退回预案**;调大后恢复真推理。
- **安全**:agent 只建议不动手(human-in-the-loop),必须基于所给数据;无网络时用内置预案兜底。

---

## 5. HMI(自研 SCADA 操作站,`app/hmi.py` + `static/index.html`)

- **看**:浏览器每 0.5s `GET /api/state` → 更新 SVG(水位高度、泵旋转、阀颜色、管线流动)。
- **控**:点 AUTO/MAN、START/STOP → `POST /api/control` → MQTT `control/op` → sim 把设备从自动切手动。这就是"按钮真能控制设备"。
- **报警**:声音警笛(Web Audio)+ 红灯 + ACK/消音按钮 + 带时间戳的 AI 建议列表。

---

## 6. 云 + 多路报警(LO3 数据通路 / LO6 边缘vs云)

- **ThingsBoard Cloud**:`tb_bridge` 把遥测 + AI故障转发上云;云端用 **Alarm rule**(`fault_active equal 1`)生成正式告警。
- **Telegram**:`telegram_notify` 订阅同一故障流,把 Claude 建议推到手机。
- 三路报警 = 真实工业的分级通知:现场操作员(HMI)、值班手机(Telegram)、管理层(云平台)。

## 8. 演示流程(demo day)

1. 打开 HMI(8090),正常运行(FILLING→TREATING→DISCHARGING 循环);
2. ALARMS 标签注入一个故障;
3. 60s 内:工艺图异常(pH变红/流量掉零)+ 声音报警 + AI 建议;同时 ThingsBoard 弹告警、Telegram 收推送;
4. 四个故障各演一遍;清除时点 clear。

> 备份:`python -m app.demo_local` 无需 Docker 即可跑全流程,演示日 Docker 出问题时兜底。
